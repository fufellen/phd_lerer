# vibr_t - программа VIBR_T А.М. Лерера (метаповерхности с ПНЧ) и её Python-порт

## Происхождение

Полные C-исходники программы **VIBR_T** получены 2026-07-04 от Льва Яковлева
(папка Obsidian `PhD/.../from_Lerer/от Лерера/Лев Яковлев/Программа попытка
отследить баг`). Это программа, которой считалась статья «Лерер АМ_1»
(оптические поглощающие метаповерхности с плазмонными наночастицами):
объёмное интегро-дифференциальное уравнение + тензорная функция Грина
слоистой среды + базис Галёркина (метод статьи Лерера 2012 г.,
DOI 10.1134/S106422691210004X), СЛАУ решается через Intel MKL.

Это **другая программа**, нежели `Полиномы 7_1/` в корне репозитория
(та - слой/решётка методом коллокации); общие у них только точка входа
`VIBR.c` по стилю, `Vvod.c` и библиотека `newLIb`.

## Состав папки

- `src_c/` - C-исходники программы как получены (перекодировка не делалась):
  ядро солвера (`VIBR.c`, `F_df.c`, `D_N_PER.c`, `EXTERN*.c`, `MLAP.C`,
  `RECUR*.c`, `BASIS.c`, `MATR*.c`, `MEMORY.c`, `Vvod.c`, ...) и
  материальная библиотека (`Au_.c`, `Ag_.c`, `Si_.c`, `ZnO_.c`, `EPS_MET.c`,
  `AZO_.c`, `SiO2_HfO.c`, `VO2_cold.c`, `VO2_hot.c`, `BIBL_mat.c`), плюс
  файлы проекта Visual Studio. Артефакты сборки (`Debug/`, `Release/`,
  `*.obj`, `*.ncb`, `*.sdf`), Origin-проекты `*.OPJ` и `*.docx`-заметки
  сознательно оставлены только в Obsidian.
- `inputs/` - входные файлы программы (несмотря на расширение `.c` это
  ДАННЫЕ в формате `/ комментарий / числа`): `A.c`, `A1.c`, `ANO.c`,
  `AP.c`, `AP2.c`, `AP3.c`, `dt.c`, `dt2.c`, `Av.c`. В актуальном формате
  Vvod.c на слой/вкладыш читается тройка `is comp CONCEN` (номер материала,
  номер материала композита, концентрация).
- `reference_outputs/` - эталонные выводы самой программы Лерера:
  `'a` (полный прогон: геометрия статьи dx=dy=500 нм, цилиндр 300 нм,
  слои n=1.77/1.45, таблицы R, T, delta P от 200 нм) и `Release_'a`
  (короткий фрагмент из Release-папки). Использовать как референс для
  кросс-проверки нашего RCWA-солвера.
- `data/` - табличные оптические константы, автоматически извлечённые из
  C-исходников скриптом `scripts/extract_c_tables.py` (не перепечатка!).
- `scripts/` - Python-порт (см. ниже).
- `results/` - CSV/PNG результаты скриптов.

## Что портировано на Python

| Скрипт | Что делает |
| --- | --- |
| `extract_c_tables.py` | парсит массивы `Lam/n/k`, `xi/fs`, `xii/fss` прямо из C -> `data/*.csv` |
| `vibr_materials.py` | порт материальной библиотеки: `libmat(is, lam)` с семантикой C-интерполяций (Cu=1, Ag=2, Au=3, ZnO=4, Si=5, SiO2=6, HfO2=7, AZO=8, VO2=20/21) |
| `vibr_composite.py` | порт `COMPOSITE` (формула Клаузиуса-Моссотти), `COMPOSITE_3` (2 типа НЧ; есть версия bug-for-bug и исправленная), `COMPOSITE_me` |
| `validate_port.py` | валидация порта (узлы таблиц, интерполяция, паритет с `composite_ema`, демонстрация бага, Фрёлих) |
| `compare_constants_lerer_vs_jc.py` | константы Лерера vs Johnson & Christy 1972 для Au/Cu |
| `rerun_fig2_lerer_constants.py` | ПЕРЕСЧЁТ рис. 2 статьи (RCWA 1D, TE) на константах Лерера - закрывает УФ-вопрос Au/Cu |

Ядро электродинамического солвера (Галёркин + функция Грина + MKL,
`F_df.c`/`MLAP.C`/`D_N_PER.c` и др.) на Python НЕ портировано - это
отдельный большой проект; для той же физической задачи в репозитории уже
есть независимый валидированный солвер `grating_2d_rcwa/` (1D и 2D RCWA).
C-исходники ядра сохранены здесь полностью для воспроизводимости и
будущего порта.

## Запуск

```powershell
cd vibr_t\scripts
python extract_c_tables.py     # регенерация data/ из src_c/
python validate_port.py        # все проверки должны быть OK
python compare_constants_lerer_vs_jc.py
python rerun_fig2_lerer_constants.py
```

Зависимости: `pip install -r requirements.txt` (numpy, matplotlib).

## Ключевые находки (статус: validated, см. validate_port.py)

1. **Баг в `COMPOSITE_3` (VIBR.c)** - мнимая часть числителя формулы КМ
   собрана из действительной: `v1 = 2 * Re_sum;` вместо `2 * Im_sum`.
   Даже при C2=0 функция не сводится к корректной `COMPOSITE`; на
   Au/C=10%/554 нм ошибка eps_eff ~76%, |Im eps_eff| занижена в ~7 раз
   (поглощение композита почти обнулено). Вероятно, это и есть баг,
   который искали (папка называлась «Программа попытка отследить баг»;
   расчёт Яковлева с 2 типами НЧ шёл через эту функцию).
2. **У таблицы Cu нет УФ-данных** (`EPS_MET.c`: 517-3900 нм). Ниже 517 нм
   локальная интерполяция линейно экстраполирует по точкам 517/530 нм;
   Re eps становится положительной ниже ~416 нм (на 250 нм Re eps=+9.3) -
   медь в УФ в программе ведёт себя как диэлектрик с потерями.
   **Пересчёт рис. 2 подтвердил**: с константами Лерера средний УФ P:
   Au 0.782 > Cu 0.537 (порядок как в статье); с Johnson & Christy:
   Cu 0.828 > Au 0.812 (наше прежнее «расхождение»). УФ-порядок Au/Cu
   на рис. 2 статьи - артефакт отсутствующих УФ-данных меди, а не физика.
3. Интерполяция «плотных» таблиц (`Au_.c` и др.) из-за описки
   (`Lam[m]` вместо `lam`) кусочно-постоянная, не линейная; на целых
   длинах волн (все входные файлы) разницы нет.
4. Верхний клэмп окна интерполяции в `EPS_MET.c` использует заявленную
   ёмкость массива (44/86/85), а заполнено на точку меньше - за правым
   краем таблицы C-код читал неинициализированную память.
5. Снимок программы в присланном виде **не собирается**: весь `BIBL_mat.c`
   закомментирован (`libmat` линкуется из прекомпилированной
   `newlib+mat.lib`), а линковка старых `newlib+mat.lib`/MKL падает на
   современном MSVC (`___iob_func`, ...) - потому и требовалась VS2013.
   В `src_c/` лежит как есть; Python-порт от сборки C не зависит.
6. Материалы 9-19 из диспетчера (`HfN_`, `TiN_`, `ZrN_`, `ITO_`, `GZO_`,
   `AZO_wt_*`, ...) в снимке отсутствуют (нет исходников функций) -
   в porte недоступны; HfN/TiN/ZrN для рис. 5 статьи есть в
   `composite_ema/` из открытых источников.

## Конвенция знака

Во всей папке используется конвенция Лерера e^{+i w t}: `Im eps <= 0` для
пассивной среды. Перед передачей в `grating_2d_rcwa`/`tmm` (фотонная
конвенция e^{-i w t}) берётся комплексное сопряжение.
