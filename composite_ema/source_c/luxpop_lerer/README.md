# Материальная база Лерера (luxpop) — оригинальные C-исходники

Байт-в-байт копии оптической базы диэлектрических проницаемостей из программы
Лерера А.М. и др. ([2–4] статьи), присланной 05.07.2026 (папка Obsidian
`.../от Лерера/От Лерера 05.07.2026/newLIb+mat`). Это те самые «таблицы
программы», которые в статье «Влияние модели эффективной среды и источника
оптических констант…» сопоставляются с измерениями Джонсона и Кристи; их
первоисточник — база **luxpop** [11], а таблицы нитридов/оксидов оцифрованы или
взяты из отдельных измерений (см. ниже). Файлы **не менялись** — сохранены как
исторический источник и как вход для `libmat()`, вызываемой из `COMPOSIT.c`.

`BIBL_mat.c` (диспетчер `libmat()`) взят из другой папки Лерера
(`.../от Лерера/Лев Яковлев/Программа попытка отследить баг/BIBL_mat.c`) — это
то самое `libmat()`, которое в `../COMPOSIT.c` было помечено как «не найдено в
доступных исходниках». Его комментарии в оригинале в кодировке Windows-1251
(в UTF-8 читаются как «кракозябры»); карта индексов ниже восстановлена по телу
`switch`.

## Соглашения

- **Знак**: `eps = eps' - i*eps''`, у пассивной среды `Im(eps) <= 0`. Функции
  материалов хранят `eps''` положительным, но возвращают `*Im_eps = -eps''`
  (см. хвост `15_HfN.c`), т.е. уже в этой конвенции — совместимо с
  `COMPOSITE()`/`BRUGGEMAN()`/`COMPOSITE_MLWA()` в `../COMPOSIT.c`,
  `../COMPOSITE_MODELS.c`.
- **Единицы**: длина волны приходит через глобальную `f`. Металлы (`Ag_`, `Au_`)
  и нитриды/оксиды (`HfN_`, `TiN_`, `ZrN_`, `GZO_`, `ITO_`, `TaN_`) работают в
  **нм** (табличный шаг 1 нм, индекс массива = длина волны). `AZO_`/`AZO_2` и
  `AZO_wt_*` переводят вход в **мкм** внутри функции (`lam/1000`).
- Диапазоны у каждого материала свои; за пределами таблицы функции печатают
  `... not defined` и/или линейно экстраполируют по крайним узлам. Важно для
  статьи: таблица **Cu начинается с 517 нм**, и ниже программа линейно
  экстраполирует — именно этот артефакт разбирается в разделе «Чувствительность
  к источнику оптических констант».

Материалы `is = 4..7, 9` (`ZnO_`, `Si_`, `SiO2_`, `HfO_`, `EPS_R`) диспетчер
вызывает, но их файлы **не входили** в присланный бандл `newLIb+mat`; их
исходники есть в других папках Лерера (`Si_.c`, `ZnO_.c`, `SiO2_HfO.c` в
`.../Лев Яковлев/Программа попытка отследить баг/`) и здесь не продублированы,
чтобы не расширять копию сверх присланной базы.

### Важно: две разные нумерации

- Диспетчер `BIBL_mat.c` использует **Au = 3, Ag = 2, Cu = 1**.
- Комментарий в `../COMPOSIT.c` и Python-порт (`scripts/composite_ema.py`)
  используют **Au = 4** (и свою нумерацию `5=HfN, 6=ZrN, 7=TiN`).

Это исторические, несовпадающие схемы. При использовании `libmat()` из
`BIBL_mat.c` действует таблица выше; Python-порт задаёт материал напрямую и от
`libmat()` не зависит.

### `EPS_MET.c` — Ag/Au/Cu

`EPS_MET.c` определяет `Ag(x,ri)`, `Au(x,ri)`, `Cu(x,ri)` (локальная
интерполяция Лагранжа, `ri=0` — Re, `ri=1` — Im). Диспетчер берёт отсюда только
**`Cu`** (для `is=1`); Ag и Au он берёт из более полных таблиц `Ag_.c`/`Au_.c`
(200–2000 нм, шаг 1 нм). Функции `Ag()`/`Au()` в `EPS_MET.c` — более старые
таблицы с меньшим числом узлов, оставлены как есть.

## Связь с расчётным кодом

- `../COMPOSIT.c` — Клаузиус-Моссотти (Максвелла-Гарнетта), формулы (1)/(5)
  статьи: `COMPOSITE()` (1 сорт), `COMPOSITE_3()` (2 сорта, исправленная).
- `../COMPOSITE_MODELS.c` — добавленные модели: `BRUGGEMAN()` (формула (4)) и
  `COMPOSITE_MLWA()` (Максвелл-Гарнетт с размерной поправкой MLWA, [9]). Обе
  берут `eps` частиц из этой базы через `libmat(n_mat, f, ...)`.
- Python-эквиваленты и проверка — `../../scripts/`.
